## Project Nature

This is a **preview tool** — a design/prototyping workbench that renders screens inside device frames. It is NOT a traditional app with user-facing routes. There are two screen authoring approaches:

### 1. MDX Content Screens (primary, in `/content/`)
- Files in `/content/**/*.mdx` are auto-discovered and rendered inside a device frame
- Route derived from file path: `/content/booking/type.mdx` → route `/booking/type`
- States defined in YAML frontmatter; toggled via CatalogPanel sidebar
- Use `<Variant state="stateName">` to conditionally render per state
- MDX components are globally available (no imports needed)

### 2. React Screens (in `src/screens/`)
- Traditional React components with scenarios + `<ScenarioSwitcher>`
- Screens grouped by section: `src/screens/{section}/{screen}/` or `src/screens/{screen}/` for standalone
- Locale files co-located in screen folders (e.g., `src/screens/prescription/scan/en.json`)
- Auto-discovered via `import.meta.glob('**/index.tsx')` — `_shared/` folders excluded

**New screens should use the MDX approach** unless React-specific logic is required.

## Conventions
- All client-facing strings use `t()` from react-i18next — no hardcoded text
- Every `<Input>` has an associated `<Label>` with matching `htmlFor`/`id`
- Lists/tables handle three states: loading, empty, populated
- Status badges: PENDING=amber, CONFIRMED=emerald, COMPLETED=sky, CANCELLED=red
- Buttons have `disabled` state during async + loading feedback
- Color-coded visuals always include a text label — never color-only
- Page layout: `min-h-svh`, `bg-background`, `p-4` minimum
- Card max width: `max-w-md` for forms, `max-w-4xl` for tables/lists
- Gap between stacked elements: `gap-6`
- shadcn/ui components added via `pnpm dlx shadcn@latest add <component>`
- No npm packages without explicit user approval
- Dependencies flow: tokens → ui → blocks → screens (never upward)
