# Screen Spec Generator

Generate structured per-screen specs that give Claude (or any developer) everything needed to build a page correctly on the first pass.

## Workflow

```
User says route or screen name
  → Phase 0: Discover project context
  → Phase 1: Gather screen context
  → Phase 2: Ask clarifying questions (AskUserQuestion)
  → Phase 3: Draft spec with ASCII layout
  → Phase 4: Present for approval
  → Phase 5: Save spec file
```

## Phase 0: Discover Project Context

**Skip if** `docs/plans/screens/project-context.md` already exists — read it instead and jump to Phase 1.

On first invocation, scan the project to understand its conventions:

1. **Project instructions** — `CLAUDE.md`, `README.md`, or similar
   - Tech stack, component library, design system, conventions
2. **Existing specs** — Glob for `docs/plans/screens/*.md`, `docs/specs/*.md`, or similar
   - If specs exist, read one to learn the project's spec format
3. **Sitemap / routing** — Find the router config:
   - React: `App.tsx`, `router.tsx`, `routes.tsx`, or files with `<Route` / `createBrowserRouter`
   - Next.js: `app/` directory structure
   - Vue: `router/index.ts`
   - Other: search for route definitions
4. **Navigation docs** — Glob for `sitemap*.md`, `siteflow*.md`, `navigation*.md`
   - Extract flow position, prev/next, guards if documented
5. **Screen data docs** — Glob for `screen-data*.md`, `screen-spec*.md`
   - Extract element tables if they exist
6. **Existing pages** — List files in the pages directory to identify layout patterns in use
   - Read 2-3 representative pages to understand structural conventions

## Phase 1: Gather Screen Context

For the specific screen being spec'd:

1. **Navigation context** — From routing config and navigation docs:
   - Flow position (step N of M, or standalone)
   - Previous/next routes and triggers
   - Back target, guards, on-success actions
2. **Existing docs** — Check if a spec or element table already exists for this route
3. **Layout pattern match** — Read `references/layout-patterns.md` from this skill
   - Match the screen to a known pattern (LP-1 through LP-15)
4. **Similar pages in codebase** — If the screen resembles an existing page, read that source
   - e.g., new list page → find and read an existing list page for structural patterns
5. **Domain knowledge** — For common screen types (settings, list, detail, form, dashboard), suggest elements based on standard UI/UX patterns from training data

## Phase 2: Ask Clarifying Questions

Use `AskUserQuestion` for ANY uncertainty. Never guess on:

- **Screen purpose** — "What is the primary action on this page?"
- **Layout pattern** — "This looks like a [pattern]. Should I use that or something different?"
- **Data source** — "Should this read from [collection/API/store]?"
- **Element ambiguity** — "The spec mentions 'filter' — dropdown, tabs, or search input?"
- **Conditional logic** — "When should [element] be shown/hidden?"
- **Navigation** — "Where should [action] navigate to?"
- **Existing page reference** — "This is similar to [page]. Should I base the layout on it?"

Ask one question at a time. Prefer multiple-choice options when possible.

**When to ask vs. when to infer:**
- **Ask** if the answer affects layout structure, data flow, or user actions
- **Infer** if the answer is obvious from the router, existing docs, or codebase patterns

## Phase 3: Draft the Spec

Read the template at `references/spec-template.md` from this skill. Adapt it to the project's conventions (component library, data layer, etc.). Every spec MUST include:

### Required Sections

**1. Header**
```markdown
# `/route/path` — Screen Name

**Source:** [requirement reference if available]
**File:** `path/to/page-file.tsx`
```

**2. Navigation Context**
```markdown
### Navigation Context

- **Flow:** [flow name] (step N of M) | standalone
- **Previous:** /prev/route (trigger)
- **Next:** /next/route (trigger) | none (terminal)
- **Back target:** /back/route
- **Guard:** [auth + preconditions]
- **On success:** [action result — API call, store update, navigation, toast]
```

**3. Elements** — Use detailed table for 5+ elements, bullet list for simpler screens.

Common element types: Text, Input, Button, Link, Image, Badge, Card, List, List Item, Tab, Dropdown, Navigation, Switch, Checkbox, Read-only, Date Input, Date Range, Textarea, Tags, Timer, Selector, Toggle, Visual, Widget, Grid, Calendar, Table, Banner, Alert, Preview, Status Indicator, Upload, Static, Email

Adapt to the project's component library (e.g., shadcn/ui, MUI, Ant Design, Chakra).

**4. States** — Check all that apply: Loading, Empty, Populated, Error, Not found, Offline. Include conditional elements (show/hide/disable logic).

**5. Data** — Reads (collections, stores, API calls, URL params) and Writes (API calls, store updates, toasts).

### Layout Reference

If the screen matches an existing layout pattern or an existing page in the codebase, add a reference:

```markdown
**Layout pattern:** LP-4 (Detail page with read-only rows) — see `path/to/similar-page.tsx`
```

This tells the implementer to read that file for structural patterns.

### ASCII Layout Preview

ALWAYS generate an ASCII layout sketch. Use box-drawing characters:

```
┌─── ┐  └─── ┘  │  ─  ├  ┤  ┬  ┴  ┼
```

Rules:
- 42-char wide outer frame
- Label each section on the right side with a comment
- Show conditional elements with a note (e.g., "conditional alert")
- Show all states inline where possible (e.g., "[Badge]" for status)
- Use realistic placeholder data relevant to the app's domain, not "Lorem ipsum"

Example:
```
┌──────────────────────────────────────────┐
│ ← Back link                              │  nav
├──────────────────────────────────────────┤
│ "Page Title"                   [Badge]   │  header
│ Subtitle text                            │
├──────────────────────────────────────────┤
│ ┌──────────────────────────────────────┐ │
│ │ Content area                         │ │  card
│ └──────────────────────────────────────┘ │
├──────────────────────────────────────────┤
│ [Action 1]  [Action 2]                   │  actions
└──────────────────────────────────────────┘
```

### Optional Sections

Add only when relevant:
- **Constraints** — Validation rules, rate limits, privacy/GDPR, accessibility
- **i18n Keys** — Only if the project uses i18n and new keys are needed

## Phase 4: Present for Approval

Show the complete spec to the user. Include the ASCII layout prominently. Ask:

> "Here's the spec for `/route`. Does this look right, or should I adjust anything?"

If the user wants changes, revise and re-present. Do NOT save until approved.

## Phase 5: Save

Save location depends on project conventions discovered in Phase 0:
- If `docs/plans/screens/` exists → save there
- If `docs/specs/` exists → save there
- Otherwise → create `docs/plans/screens/` and save there

File naming:
- Nested routes like `/book/doctor` → `book-doctor.md`
- Parameterized routes like `/items/:id` → `item-detail.md`
- Simple routes like `/settings` → `settings.md`

## Context Sources (Priority Order)

1. **Project docs** (highest) — CLAUDE.md, existing specs, sitemap/siteflow docs
2. **Existing code** — similar pages in the codebase for structural patterns
3. **Layout patterns** — `references/layout-patterns.md` in this skill
4. **Domain knowledge** — standard UI/UX patterns from training data for the app's domain

For screens not yet documented, suggest elements and flows based on common patterns for that screen type (e.g., a "settings" page typically has sections with toggles and form fields).
